## Overview
Pizzard is a top-down 2D spellcasting roguelite game built in Unity. The player fights bosses by combining elemental ingredients (Cheese, Pepperoni, Pineapple) to cast unique spells. The game relies heavily on persistent Singleton Managers to track progression and control the flow between distinct scenes (Menu, Narrative Dialogs, Shops, Boss Arenas).

## System Diagram
```
[Player Input (New Input System)] ---> [PlayerAimAndCast / PlayerController] 
                                        |
                                        v
[ElementsCombiner] <--- [PlayerEquip] (Wand Tier Defines Max Elements)
        |
        +-- Builds spell key (e.g., "queso|pepperoni") ---> [CombinationDatabase (ScriptableObject)]
        |
        v
[Instantiates Attack Prefabs] ---> Deals damage to [BossBase / PblobController]

--------------------------------------------------------------------------------------
[GameFlowManager (Singleton)]
    |--> Controls Scene Loading & TimeScale
    |--> Routes to [ShopUI], [DialogUI], [BossArenaManager]
    |--> Interacts with [SaveManager] and [ProgressionManager] for tokens/levels.
```

## Core Components

### GameFlowManager
- **Purpose:** Central state machine dictating the game loop (MainMenu -> Intro -> Shop -> PreBoss -> BossFight -> PostBoss -> Shop -> ... -> Credits).
- **Location:** `Assets/Scripts/GameFlowManager/GameFlowManager.cs`
- **Dependencies:** SceneLoader, UIManager, SaveManager, ProgressionManager.

### UIManager
- **Purpose:** Persistent canvas manager that toggles different UI panels depending on the current GameState.
- **Location:** `Assets/Scripts/UI/UIManager.cs`
- **Children:** `ShopUI`, `DialogUI`, `DeathUI`, `CharacterHPUI`, `MenuUI`.

### Player Controllers
- **Purpose:** Handles 2D physics movement, aim, casting, and taking damage.
- **Location:** `Assets/Scripts/Player/`
- **Key Scripts:** `PlayerController.cs` (Movement & Dash), `PlayerAimAndCast.cs` (Spell Execution), `PlayerHPController.cs` (Health & Death), `PlayerEquip.cs` (Wand progression).

### ElementsCombiner
- **Purpose:** Tracks selected elemental ingredients based on wand tier limits, queries the `CombinationDatabase`, and applies fatigue cost via `FatigueSystem`.
- **Location:** `Assets/Scripts/Elements/ElementsCombiner.cs`

### SaveManager & ProgressionManager
- **Purpose:** JSON-based persistent storage for Tokens, current Tier, unlocked elements, and defeated bosses.
- **Location:** `Assets/Scripts/Progression/SaveManager.cs`

### Boss Logic
- **Purpose:** Custom phase-based behaviors. `PblobController` implements a 4-phase boss battle with specific element weaknesses and attack patterns.
- **Location:** `Assets/Scripts/Pblob/PblobController.cs`

## Data Flow
1. **Progression:** User defeats boss -> `BossArenaManager` notifies `GameFlowManager` -> Proceeds to PostBoss Dialog -> Shop. At Shop, `ProgressionManager` grants tokens -> Player buys Wand upgrade -> `SaveManager` serializes.
2. **Combat:** User presses 1-3 -> `ElementsCombiner` adds string -> User clicks -> `PlayerAimAndCast` verifies against `CombinationDatabase` -> Checks `FatigueSystem` -> Instantiates Prefab -> Consumes Fatigue.

## Conventions
- **Naming:** standard C# camelCase for variables, PascalCase for classes.
- **Structure:** Modular singletons for global state, localized scripts attached to specific UI/Gameplay prefabs.
- **Initialization:** Heavily relies on `Awake()` Singletons and `FindObjectOfType` for dynamic scene references (since UI objects persist across loads).

## Technical Debt & Known Issues
- [ ] **Scene Initialization Dependency:** Boss Arenas require `GameFlowManager` to initialize properly. Direct testing of scenes requires bypass logic.
- [ ] **Camera Settings:** Virtual camera zoom/orthographic size is rigidly set and currently uncomfortable in boss arenas.
- [ ] **Boss Artificial Intelligence:** `PblobController` currently lacks wandering or dynamic physics movement; completely stationary unless attacking.
- [ ] **Scene Hygiene:** Many scenes have redundant or orphaned UI elements that shouldn't be there because `DontDestroyOnLoad` managers dynamically inject the true UI.
